üîç 5 Surprising Truths About Claude Code

1

It Doesn't Index Your Codebase (And That's a Good Thing)

The common assumption about AI coding tools is that to understand a large project, they must first index and embed the entire codebase using Retrieval-Augmented Generation (RAG). Claude Code does not do this at all.

Instead, it relies on "agentic search." It explores the codebase using the same command-line tools a human developer would, like grep, find, and glob. The AI can run a search, analyze the results, and decide to run another, more refined search, dynamically building its understanding of the project's structure and logic.

Why This Matters

Rather than working from a pre-processed, static snapshot, Claude Code learns the codebase on the fly, much like a new engineer joining your team. The primary benefit is currency: unlike a static RAG index that can become stale, this agentic search method ensures Claude is always working with the absolute latest version of the code.

Trade-off: Agentic search trades token efficiency for currency and security. While it avoids third-party indexing liabilities, it can consume significantly more context as it performs sequential roundtrips. For massive codebases, you can add semantic search via MCP vector database plugins like Milvus.

"If you're going to build a coding agent or a coding tool a year ago, you'd probably have ideas like... index the whole codebase and embed it and do this fancy like kind of RAG retrieval thing. That is not how Claude Code works. We don't do any sort of indexing. Instead, Claude kind of explores and understands the codebase... through agentic search."

4

A Simple Markdown File Acts as Its Shared Memory

AI models are inherently stateless; they don't have long-term memory between sessions. Claude Code addresses this with a deceptively simple and powerful mechanism: a Markdown file named claude.md.

When Claude Code starts in a directory, it automatically finds any claude.md file and loads it into context. This file serves as a set of standing instructions for the AI.

What to Include in claude.md

  • Instructions on how to run unit tests
  • Overview of the project's layout and key modules
  • Your team's specific coding style guide
  • Common commands and workflows
  • Architecture decisions and patterns
üí° Team Collaboration Benefit:
By checking the claude.md file into your project's version control (e.g., Git), the entire team can share the same set of instructions with Claude. This ensures that every developer‚Äîand the AI assisting them‚Äîis working from the same context, promoting consistency across the board.
# claude.md example
## Testing
Run tests with: npm test
Run e2e tests with: npm run test:e2e

## Architecture
This is a Next.js app with API routes in /pages/api
Database layer uses Prisma ORM
5

It Has a Clever Trick for Long Conversations

Anyone who has used a large language model for a complex task has run into the "context window" limit. As a conversation gets longer, the model eventually runs out of memory and has to reset.

Claude Code handles this with automatic compaction, but the real power lies in taking control with the /compact command.

Automated Convenience, Manual Control

While Claude will automatically compact the session when it gets full, recent analysis suggests the threshold is often lower than 95% (closer to 75-85%) to preserve "working memory" for reasoning. Compaction is a lossy compression‚Äîafter 2-3 cycles, nuanced details can be lost in a "telephone game" effect.

The "Document & Clear" pattern is a power-user best practice: have Claude write its progress to a progress.md file, then run /clear and start fresh using that file as the new baseline. This provides a lossless checkpoint rather than trusting lossy automatic summarization.

Preserves Momentum Maintains Context Avoids Full Reset

This is a game-changer for complex tasks. It preserves institutional knowledge within the session while keeping the context fresh and fast.

/compact

üéØ Working with Code

For Complex Changes

If you're planning a major change, such as large-scale refactoring or adding complex new functionality, always instruct Claude Code to outline a plan before implementing any changes. Encourage deeper thinking by using phrases such as:

Think hard Think deep Think longer

After reviewing Claude's proposed plan, if you agree, proceed directly with the changes. If not, request revisions to the plan.

For Smaller Changes

For smaller changes, such as simple, single-file updates or straightforward features, clearly specify the exact file you want to modify. For example:

Claude, can you increase the size of the chat input in the file "user/page.tsx"?
Claude, can you add error handling to "data_processing.py"?
Claude, please refactor the `calculate_totals` function in "invoice.py"
Best Practices

Other Best Practices

  • Regularly use the /compact command during development. This helps Claude Code remain efficient and reduces the risk of confusion in long-term projects.
  • Always be explicit about which files you intend to change and clearly state their relationships to other components.
  • Frequently run tests. Claude Code excels at validating new functionality through thorough testing, ensuring reliability and accuracy.
  • Periodically ask Claude Code to review the entire project, assessing logic consistency and identifying any redundant or inefficient code.

Want Action, Not Advice? Say "Do It"

Control Formatting by Saying What to Do
Instead of "don't use markdown," say "write 3 short paragraphs" or "return JSON."

Recommend lightweight tags to set defaults like "be proactive" vs "be cautious."
‚úì Good Prompt:

‚úó Bad Prompt:
"Be proactive."
9.
Encourage "Chain of Thought"
For complex logic, explicitly ask Claude to plan before executing.
‚úì Good Prompt:
"Before writing the code, outline your step-by-step reasoning and potential edge cases."
‚úó Bad Prompt:
"Just fix the bug immediately."

Advanced techniques used by Anthropic's own engineering team

Workflow

Multi-Claude Approach

Don't just use one agent. Anthropic engineers often run multiple Claude instances simultaneously.

  • Coder & Reviewer: Have one terminal open where Claude writes code. Open a second terminal to have another Claude review that code.
  • Git Worktrees: Use git worktree to check out multiple branches in different folders. Run a separate Claude in each to work on independent features in parallel.
Methodology

The TDD Power-Loop

Test-Driven Development is supercharged with agents. Be explicit about the steps:

  1. "Write a test for Feature X. Do not implement it yet."
  2. "Run the test and confirm it fails."
  3. "Now write the code to pass the test."
  4. "Run the test again to verify."

This prevents "hallucinated success" and keeps the agent grounded in reality.

‚ö° Automation: Custom Slash Commands

Did you know you can create your own commands? Create markdown files in a .claude/commands folder to define reusable workflows.

# .claude/commands/fix-issue.md
Please analyze and fix the GitHub issue: $ARGUMENTS.
1. Use `gh issue view`...
2. Search codebase...
3. Fix and test...

Now you can just type /project:fix-issue 123 to trigger a complex, multi-step engineering protocol.

Advanced Control

Hyper-Control with Hooks

Hooks aren't just for blocking rm -rf. They are the primary mechanism for enterprise governance and workflow automation. You can trigger scripts at lifecycle events like PreToolUse, PostToolUse, or OnSessionStart.

Safety & Governance:
  • Enforce read-only access to specific directories
  • Require human confirmation for production config changes
  • Log all tool invocations for compliance audit trails
Workflow Automation:
  • Auto-run linters/tests after every code modification
  • Inject team-specific style guides into the context
  • Validate outputs against schemas before execution

Stop trying to do everything in one context window. Delegate.

Architecture

The Mental Model: Isolation

Subagents aren't just for delegation; they solve context pollution. By spawning a specialist with a fresh context window, you prevent irrelevant exploration noise from drowning out the signal in your main session.

This architectural pattern ensures isolated memory and single-purpose focus. When a subagent finishes, it returns only the result‚Äînot thousands of tokens of trial and error.

How-To

Creating Subagents

Two ways to spawn your team:

  • Interactive: Type /agents and let Claude design a specialist for you.
  • Declarative: Version-control your team via .claude/agents/*.md files, defining specific tools and guidelines.

The Killer Combo: Subagents + MCP

Combine them for secure, powerful workflows. A subagent can be restricted to only use specific MCP tools.

# .claude/agents/database-admin.md
description: A cautious database specialist
tools: ["postgres-mcp", "read-only"]

You are a DBA. You verify every SQL query twice. You NEVER run DELETE without explicit confirmation.

This agent can't accidentally edit your frontend code‚Äîit doesn't even have the tools to see it.
